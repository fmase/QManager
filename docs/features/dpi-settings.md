# DPI Settings (Video Optimizer + Traffic Masquerade)

CGI: `network/video_optimizer.sh` · Hooks: `use-video-optimizer.ts`, `use-cdn-hostlist.ts`, `use-traffic-masquerade.ts` · Types: `video-optimizer.ts` · Reboot: No

- **Unified UI — "Traffic Engine".** Both modes share ONE page at `/local-network/traffic-engine` (a dedicated first-class Local Network sidebar entry, icon `MonitorPlay`, lifted out of the "Local Network Settings" collapsible). A tabbed mode control (Video Optimizer | Masquerade) is a VIEW selector; deep-link the mode with `?mode=video|masquerade`. The page is a System-Settings-shape two-column grid (the CDN hostlist in column 2 stretches to column 1's height; column 2 is empty on the Masquerade tab). Components live under `components/local-network/traffic-engine/`: `traffic-engine.tsx` (composer + tab crossfade), `engine-status-card.tsx` (live stats + idle-only remove footer), `engine-enable-row.tsx` (shared enable switch), `bypass-hotspot-row.tsx` (shared Bypass Hotspot switch — see below), `engine-check-row.tsx` (shared Verify Service / Test Injection layout), `video-optimizer-panel.tsx`, `masquerade-panel.tsx`, `cdn-hostlist-card.tsx`, `result-alert.tsx` (shared tinted result alert), `engine-advanced.tsx` (exports `EngineRemoveRow`, hosted in the status-card footer), `engine-onboarding.tsx`. The old per-feature components (`components/local-network/video-optimizer/*`, `traffic-masquerade/*`, `service-stats.tsx`) and the first-redesign components (`engine-status-strip`, `engine-mode-toggle`, `throughput-hero`, `mini-sparkline`, `live-stat-tile`, `video-optimizer-mode`, `masquerade-mode`, `cdn-hostlist-manager`) are superseded and orphaned.
- Routes: canonical `/local-network/traffic-engine`. Old `/local-network/video-optimizer` → `…/traffic-engine`, `/local-network/traffic-masquerade` → `…/traffic-engine?mode=masquerade`, `/local-network/dpi-masking` → `…/traffic-engine` (all redirects).
- **Mutex surfaced as takeover, not a warning.** The segmented control expresses one-mode-at-a-time structurally; the old "the other feature is active" alerts are gone. Enabling a mode while the other owns the engine opens a takeover confirm, then a single `saveSettings({enabled:true})` does the switch (backend auto-disables the other mode on save — see below). After any save the UI refreshes BOTH hooks so the status strip reflects the new engine truth. The throughput hero shows a green state dot in EITHER running mode (green = engine healthy); mode is conveyed by label, never colour. The packets/s rate + sparkline are derived client-side from per-poll `packets_processed` deltas (1s poll cadence); no new backend field.
- Binary: `nfqws` from zapret, installed to `/usr/bin/nfqws` on demand by `qmanager_dpi_install` (arch-detect → fetch `openwrt-embedded.tar.gz`). State files: `/tmp/qmanager_dpi_install.{json,pid}`.
- **Single shared nfqws on queue 200** — VO and masquerade are mutually exclusive modes of ONE process: single PID (`/var/run/nfqws.pid`). Backend enforces mutex in `save`/`save_masquerade`; init.d checks masquerade first, then VO.
- **nft rules are persistent**, shipped as `/etc/nftables.d/12-mangle-qmanager-dpi.nft` (chain `mangle_postrouting_qmanager_dpi`, `oifname "rmnet*"`, `queue num 200 bypass`). fw4 sources the file on every load/reload, so rules survive `fw4 reload` (VPN toggles, port-forward edits, mwan3 ipset refreshes, etc. — all of which used to silently wipe the runtime-injected rules). The `bypass` flag means rules are safe to leave permanent even when nfqws is not running. The init.d script never touches nftables — it only manages the daemon. `dpi_helper.sh` no longer has `dpi_insert_rules` / `dpi_remove_rules`. Uninstall removes the .nft file and drops the live chain.
- Modes: VO = SNI split (`split2`) + QUIC desync, filtered by `--hostlist`. Masquerade = fake TLS ClientHello with spoofed SNI (default `speedtest.net`), all traffic.
- Hostlist: `/etc/qmanager/video_domains.txt` (active) + `video_domains_default.txt` (immutable). Hostlist CGI supports GET `?section=hostlist`, POST `save_hostlist`, POST `restore_hostlist`.
- GET handlers gate live stats on UCI `enabled` to avoid cross-mode contamination. Kernel check: `dpi_check_kmod()` reads `/proc/config.gz` for `CONFIG_NETFILTER_NETLINK_QUEUE=y`.
- Boot persistence: enabling either mode → init.d `enable`; disabling → `disable` only if BOTH are off. Uninstall always `disable`s.
- Deps: `libnetfilter-queue`, `libnfnetlink`, `libmnl`, full `curl`, NFQUEUE kernel support.
- **Bypass Hotspot row** (`bypass-hotspot-row.tsx`) is a friendly alias over the **global TTL/HL** state — NOT part of nfqws/the engine. It reuses `useTtlSettings` + `network/ttl.sh` (no new backend). The row is **presentational**: the `useTtlSettings` instance lives in the composer (`traffic-engine.tsx`) and is passed down as the `ttl` prop to both mode panels, which forward it to the row. Owned in the composer (which persists across tab switches) rather than in the row (whose panel unmounts on every Video⇄Masquerade flip) so it never refetches/flashes a spinner mid-card; `ttl.isLoading` is folded into the page's skeleton gate so its first load is covered too. Rendered identically beneath the enable switch in BOTH panels (mirrors `EngineEnableRow` markup + micro-states so the two adjacent switch rows never drift). Ownership is **value-based** because the canonical file `/etc/firewall.user.ttl` is written by both `ttl.sh` AND `qmanager_profile_apply` (a SIM profile's TTL shows up in the same global read, with no provenance marker): `0/0` → OFF + interactive (toggle on ⇒ `saveTtlHl(64,64)`); `64/64` → ON + interactive (toggle off ⇒ `saveTtlHl(0,0)`); any other active value → ON + **disabled** (external owner — TTL page or SIM profile — manage it there). i18n keys: `traffic_engine.bypass_*`.
